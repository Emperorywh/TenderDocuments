# ADR-002 采用模块化单体

- 状态：Accepted
- 日期：2026-07-22

## 背景

首版既要保证领域边界清晰（项目、文件、分析、风险、复核、报告等互不越界），又要
避免微服务带来的分布式事务、服务发现与多部署复杂度。团队规模与运维预算有限。

## 决策

采用模块化单体（modular monolith）：单一代码库内按领域模块组织，每个模块使用
domain / application / infrastructure / api 分层；模块间通过应用服务、端口、领域
事件与事务 Outbox 协作，禁止跨模块直接读写 ORM Model（PLAN.md 第 3 节）。

## 后果

- 收益：边界清晰且事务简单（单库），未来可沿模块缝拆分；
- 代价：需要架构测试（A-006）长期守护边界，防止模块退化成大泥球；
- 不引入 Kubernetes、独立消息总线或微服务（SPEC.md 第 3.3 节）。

## 重评条件

当某模块的负载或发布节奏显著异于其他模块，且模块边界已通过架构测试长期保持稳定，
可重新评估是否将该模块拆为独立部署单元。
