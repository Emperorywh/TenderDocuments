# ADR-005 Celery 不承担业务状态机

- 状态：Accepted
- 日期：2026-07-22

## 背景

Celery 的任务状态（PENDING/STARTED/SUCCESS 等）受 broker 与 Result Backend 影响，
不稳定也不可审计；若业务状态机依赖它，会出现“任务重试导致状态回退”或“状态丢失”。

## 决策

Celery 仅负责任务投递与执行；业务状态（运行、任务、完整性）由 PostgreSQL 中的显式
字段与状态机表达。Worker 以幂等键领取任务并原子提交结果，Celery 状态不作为业务依据
（SPEC.md 第 5.4 节）。

## 后果

- 收益：状态转换集中、可测试、可恢复，与队列实现解耦；
- 代价：需要自建幂等领取、心跳与卡死扫描（阶段 D）；
- 可替换 Celery 而不重写状态机。

## 重评条件

当需要更细粒度的任务编排（如 DAG、子任务依赖）且 Celery 方案成本过高时，重新评估
任务框架，但业务状态仍须留在数据库。
