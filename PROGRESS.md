5. MVP 技术架构文档：模块边界、数据流、状态流和部署设计细节。
6. 原始需求分析：背景材料；与以上基线冲突时不作为最终实施依据。

需求冲突遵循 `SPEC.md` 中定义的优先级。进度冲突遵循“实际代码与运行证据 > Git 状态 > 本文件记录 > 计划性描述”。

## 4. 已确认的架构约束

- 本项目是全新系统，不兼容旧代码或旧数据，不保留 legacy、灰度、fallback 或 deprecated 分支。
- 前端采用 React 19、TypeScript、Vite；后端采用 Python 3.12、FastAPI、SQLAlchemy 2、Celery。
- 后端采用模块化单体，API 与 Worker 同库、不同进程；领域模块内部按 `domain / application / infrastructure / api` 分层。
- PostgreSQL 是业务状态的唯一事实来源；Redis、Celery 和前端缓存不得成为业务状态事实来源。
- 首版是单实例、单工作区且无应用鉴权，不建设用户、组织、角色、令牌、RBAC、多租户字段或 RLS。
- 原始文件不可变；解析、OCR、模型、Prompt、Schema、规则和报告产物必须可追溯、可版本化。
- AI 输出先进入候选结果，经过 Schema、证据和领域规则校验后才能晋升正式结果。
- 正式事实、要求和风险必须能回查文件、页码、原文及坐标；报告使用不可变快照。
- 不允许跨模块直接修改其他模块 ORM 数据，不允许 API 路由或 Celery Task 承担领域逻辑。
- 前端服务端状态由 TanStack Query 管理，不在全局 Store 中复制业务事实。

主业务数据流：

```text
项目与文件
→ 异步解析
→ 结构化抽取
→ 风险分析
→ 人工复核
→ 不可变报告
→ PDF 导出
```

## 8. 每次开发后的维护规则

每完成或暂停一个原子任务，必须在同一轮修改中更新本文件：

1. 更新顶部时间、当前阶段、当前任务、下一任务、完成数量和阻塞项。
2. 在“当前真实状态”中只写已落盘的交付物，不把计划写成结果。
3. 在“验证证据”中记录任务 ID、命令或步骤、固定输入、期望结果、实际结果和日期。
4. 如有失败，保留失败事实和待处理原因，不用“基本完成”“应该可用”等模糊措辞。
5. 如架构决策发生变化，先更新对应 ADR、`PLAN.md` 或 `SPEC.md`，再同步本页摘要。
6. 如出现未提交变更，明确哪些是用户原有变更、哪些是当前任务变更，禁止覆盖或回退无关内容。
7. 任务只有满足 `TASKS.md` 第 16 节完成判定后才能计入完成数量。
8. 每次修改后检查耦合、技术债、架构一致性和 AI 可维护性；发现模块边界不合理时先重构边界。
