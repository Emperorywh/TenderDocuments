## 1. 任务拆分与完成规则

本清单以“一个任务、一个主要责任、一个可重复验证结论”为拆分原则。任务可以依赖前置交付物，但完成判定不得依赖尚未实现的后续功能。

- 每个任务只产生一个主要交付物或证明一个明确性质；
- 数据模型、领域规则、应用用例、基础设施适配器、API 契约和界面分别建任务；
- 正常路径、关键失败路径、幂等、恢复和可观测性分别验证，不用一个“大联调通过”替代；
- 依赖未完成时不得用临时 Mock 绕过核心领域行为；
- 外部 DeepSeek 的录制响应验证与受控真实调用验证分为不同任务；
- 前端自动验证只运行类型检查、生产构建和组件测试，不自动启动浏览器；
- 新增和修改代码必须补充必要的多行简体中文注释；
- 禁止为未来用户、组织、角色、Token、RBAC 或多租户预埋兼容分支；
- 每个任务完成后检查耦合、技术债、架构一致性和 AI 可维护性。

状态约定：

- [ ] 待办；
- [x] 交付物存在、独立验证通过且验证证据已记录。

每个任务的验证证据至少记录：任务 ID、验证命令或步骤、固定输入、期望结果、实际结果和日期。表中“独立验证”是最低验证要求，不是可省略的说明。

## 14. 每任务架构检查

以下检查适用于每个任务，不是可以在阶段末一次性补做的任务：

| 检查项 | 通过标准 |
|---|---|
| 模块耦合 | 不存在跨模块 ORM 写入、反向依赖或跨层调用 |
| 状态事实 | 业务状态不依赖内存、Redis、Celery 或前端全局 Store |
| 单一职责 | 路由、Task、Service、Repository 和组件没有承担多个业务原因 |
| 重复逻辑 | 状态、证据、错误、版本和项目归属规则只有一个权威实现 |
| 显式行为 | 不存在隐式项目归属、隐式模型选择或隐式状态转换 |
| 首版边界 | 不引入用户、组织、角色、Token、RBAC、租户字段或 RLS |
| 证据链 | 正式事实、要求和风险均可回查原文 |
| 版本化 | 解析器、OCR、模型、Prompt、Schema 和规则可追溯 |
| 可测试性 | 新业务规则能脱离框架独立测试 |
| AI 可维护性 | 模块 README、用例名称、类型和错误码足以推导行为 |
| 技术债 | 不使用临时 patch、兼容分支、重复 DTO 或静默 fallback |

## 15. 执行顺序与并行边界

阶段主依赖为：

    A 工程基线
    → B 项目边界
    → C 文件版本
    → D 异步底座
    → E 解析证据
    → F 知识与模型网关
    → G 领域分析
    → H 复核与报告
    → I 桌面端
    → J 可观测性与数据生命周期
    → K Linux 部署
    → L 验收交付

同一阶段内只有依赖列已经满足的任务可以并行。前端任务只能在对应 OpenAPI 契约稳定后开始，不得手写重复 DTO。可观测性、备份和部署任务可以在其依赖满足后提前进行，不必等待全部界面完成。

## 16. 完成判定

一个任务只有同时满足以下条件才可标记为已完成：

- 单一交付物存在且位于正确模块；
- 表中独立验证已经执行并通过；
- 验证证据已记录且他人可以重复；
- 相关错误和边界条件可见；
- 必要测试、文档和多行简体中文注释已补充；
- 未引入跨层耦合、重复逻辑、隐式状态、身份权限残留或魔法常量；
- 对应 SPEC.md 要求可以被人工验收；
- Git 变更中没有无关格式化或覆盖用户已有代码。
