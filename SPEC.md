# 四川建筑施工招标文件智能解读与投标风险提示系统 MVP 需求规格

> 规格版本：V1.0  
> 状态：需求访谈后实施基线  
> 日期：2026-07-22  
> 适用范围：四川省房屋建筑和市政工程施工招标项目  
> 验收方式：用户依据本文功能清单人工验收

## 1. 文档地位

本文是首版 MVP 的可实施需求规格，依据以下材料形成：

- 《四川建筑施工招标文件智能解读与投标风险提示系统-需求分析.md》；
- 《四川建筑施工招标文件智能解读与投标风险提示系统-MVP技术架构.md》；
- 2026-07-22 需求访谈结论。

发生冲突时按以下优先级处理：

1. 本文记录的访谈结论；
2. MVP 技术架构文档；
3. 原需求分析文档。

需求分析文档中 NestJS、Prisma 等旧技术建议失效。后端以 Python 技术架构为唯一基线。

## 2. 产品目标

用户上传四川房建或市政施工招标文件包后，系统生成一份结构化、可复核、可追溯的风险报告，覆盖：

- 项目权威事实；
- 资格要求；
- 否决投标风险；
- 评分项；
- 关键时间节点；
- 材料准备清单；
- 技术标编制约束；
- 报价与最高投标限价提示；
- 合同与履约风险提示；
- 文件内部及版本间冲突；
- 解析覆盖率、失败页和无法判断项。

每条正式结论必须能够回到原始文件版本、页码、原文和页面坐标。系统只辅助阅读和准备，不代替业务、法律或造价人员作最终判断。

## 3. MVP 范围

### 3.1 范围计算规则

首版范围固定为：

> 原需求文档中除用户、组织、鉴权和权限外的全部 P0 功能 + 批量风险复核 + PDF 报告导出。

除上述三项明确追加能力外，原需求中的其他 P1、P2 功能不进入首版。

### 3.2 支持范围

- 地域：四川省；
- 行业：房屋建筑工程、市政工程；
- 阶段：施工招标投标准备阶段；
- 输入格式：原生 PDF、扫描 PDF、DOCX；
- 输入内容：招标公告、招标文件、澄清、补遗、延期通知和常规附件；
- 文件组织：一个项目包含多个逻辑文件，每个逻辑文件包含多个不可变版本；
- 容量：单项目全部有效输入合计不超过 500 页；
- 输出：桌面端在线报告、不可变报告快照、PDF 报告；
- API 消费者：仅本系统前端，不承诺第三方公共 API；
- AI：通过 DeepSeek 外部 API 调用；
- 部署：单台 Linux 服务器上的容器化部署。

单文件大小、项目文件数量和项目总字节数必须通过部署配置限定；页数是业务验收的硬限制，其他限制不得写死在领域代码中。

### 3.3 明确不做

- 自动生成整本投标文件；
- 判断具体企业已经满足或不满足资格；
- 接入企业证照、人员、业绩和财务资料库；
- 投标文件证据匹配与提交前终检；
- 中标概率预测；
- 具体报价推荐或完整组价；
- CAD、BIM 和完整施工图深度审查；
- 确定性法律意见；
- 招标平台自动登录、下载、CA 操作或自动提交；
- 应用内用户、组织、注册登录、会话、角色权限和租户隔离；
- 不同投标人协同编标；
- 公路、水利、铁路、电力、矿山等其他专业工程；
- 移动端适配；
- Word、Markdown 报告导出；
- 企业 Logo、自定义封面和品牌模板；
- 日程提醒或日历同步；
- 项目结构复制；
- 材料负责人、完成状态和内部协作管理；
- 合同风险独立分类页面；
- 通用低代码规则平台；
- Elasticsearch、独立向量数据库、Kafka、Kubernetes 和微服务；
- 私有化大模型或本地大模型推理；
- 旧代码、旧数据和旧 API 兼容；
- 自动化发布质量门禁。

### 3.4 P1/P2 特例

以下能力虽然在原需求中不是 P0，但经访谈明确进入首版：

- 风险批量复核；
- PDF 报告导出。

原需求中的 FR-AUTH 系列、组织管理、成员管理、角色权限、登录会话和租户隔离不进入首版。首版采用单实例、单工作区模式，所有能够访问应用入口的操作人员具有相同业务能力。

## 4. 首版运行边界

### 4.1 单工作区模式

- 一套部署实例只服务一个业务工作区；
- 不建立 Organization、User、OrganizationMembership、Role、Session 或 Token 数据模型；
- 不提供注册、登录、退出、成员管理、角色管理或权限配置功能；
- API 不要求 Access Token、Refresh Token、Cookie 会话或用户身份请求头；
- 不实现 RBAC、项目级授权、租户上下文、organization_id 或 PostgreSQL RLS；
- 所有能够访问应用入口的操作人员具有相同业务能力；
- 首版数据不得用于承载多个需要相互隔离的客户或组织。

### 4.2 部署访问边界

- 因应用本身不鉴权，生产入口必须部署在受控内网、VPN 或等价的可信网络边界内，不得直接暴露到公共互联网；
- Nginx 只暴露业务所需端口，对象存储、PostgreSQL、Redis、Worker 和内部管理端口不得公开；
- 对象默认私有，只通过短期授权地址访问；
- 对象键不得包含可猜测的原始文件名路径；
- TLS、Secret 管理、日志脱敏和备份保护不因移除鉴权而降低；
- 网络边界属于部署约束，不在应用领域层模拟用户或权限。

### 4.3 数据与操作边界

- Project 是业务数据的顶层归属边界，项目内文件、分析、风险、复核和报告必须通过 project_id 显式关联；
- Worker 必须校验任务、运行和项目之间的关联，不信任消息中的孤立 ID；
- DeepSeek 请求缓存不得在项目间复用原文或模型结果；
- 关键业务操作保留不可变记录，记录 request_id、时间、动作、资源 ID 和结果，不记录不存在的用户或角色；
- 人工复核记录表达“发生了人工动作”，首版不声明具体人员身份；
- 未来如引入用户或多租户，必须作为新版本重新设计身份边界和数据迁移，不为首版预埋兼容分支。

## 5. 核心业务流程

### 5.1 主流程

1. 操作人员创建项目。
2. 操作人员上传多个 PDF、扫描 PDF 或 DOCX 文件。
3. 系统完成格式、安全、大小、页数和 SHA-256 校验。
4. 操作人员确认文件类型、发布日期、生效顺序及替代关系。
5. 操作人员发起一次分析运行。
6. Worker 完成标准化 PDF、逐页解析、OCR、章节识别和条款切分。
7. 系统全量抽取项目事实、资格、否决、评分、时间和其他要求。
8. 系统执行确定性规则、DeepSeek 语义分析及高风险独立验证。
9. 操作人员查看覆盖率、结果和证据，在抽屉中定位原文。
10. 操作人员进行单条或批量复核。
11. 满足发布条件后生成不可变报告快照和 PDF。
12. 新有效文件进入时，历史报告内容保持不变但立即标记过期。

### 5.2 文件接入状态

文件接入与分析运行是两个独立状态机，禁止用一个字段同时表达上传和分析状态。

DocumentVersion 状态：

    UPLOADING
    → STORED
    → VALIDATING
    → READY
    → PROCESSING
    → PROCESSED

异常终态：

    REJECTED
    UPLOAD_FAILED
    PROCESSING_FAILED

页面具有独立状态：

    PENDING
    NATIVE_TEXT_READY
    OCR_REQUIRED
    OCR_READY
    FAILED

### 5.3 分析运行状态

AnalysisRun 状态：

    DRAFT
    → QUEUED
    → PARSING
    → EXTRACTING
    → ANALYZING
    → VERIFYING
    → REVIEW_REQUIRED
    → READY
    → PUBLISHED

可从活动状态进入：

    CANCEL_REQUESTED → CANCELLED
    FAILED

PUBLISHED 在新有效文件、规则版本、Schema 版本或主动重分析进入后变为 OUTDATED。

完整性使用独立字段表达：

- COMPLETE：页面、任务、关键章节和分值闭合检查均完成；
- INCOMPLETE：存在失败页面、缺失附件、关键章节缺失、分值不闭合、失败任务或过期输入。

分析阶段、失败状态和完整性不得混用。

### 5.4 任务状态

AnalysisTask 状态：

    PENDING → DISPATCHED → RUNNING → SUCCEEDED
                             ├→ RETRY_SCHEDULED
                             ├→ FAILED
                             └→ CANCELLED

每次尝试单独保存 TaskAttempt。Celery 状态不是业务状态来源，PostgreSQL 是唯一事实来源。

### 5.5 风险复核状态

    DETECTED
    → NEEDS_REVIEW
    → CONFIRMED / MODIFIED / REJECTED
    → RESOLVED

- 系统原始结果不可被人工结果覆盖；
- MODIFIED 必须保存修订值和原因；
- REJECTED 必须保存驳回原因；
- 批量操作支持 CONFIRMED 和 REJECTED；
- 涉及具体字段修订时必须逐条处理；
- 批量接口返回每条记录的成功或失败结果，不得静默部分成功。

## 6. 功能需求

### 6.1 应用入口

- 首版进入应用后直接展示项目列表，不提供登录页或会话恢复流程；
- 前端不得生成、保存或发送 Access Token、Refresh Token、用户 ID、组织 ID 或角色；
- 后端不得要求身份 Cookie、Bearer Token 或自定义身份请求头；
- API 健康检查与业务接口仍必须区分，内部基础设施端口不得经前端入口暴露；
- 生产部署必须满足第 4.2 节的可信网络边界。

### 6.2 关键操作记录

- 上传、查看、下载、删除、分析、取消、重试、复核和发布等关键操作写入不可变操作记录；
- 操作记录至少包含 request_id、动作、资源类型、资源 ID、时间、结果和错误码；
- 操作记录不得包含密码、Token、签名 URL、密钥或完整招标原文；
- 首版没有用户身份，操作记录不得使用默认账号或伪造操作者填充；
- 业务状态机不得依赖操作记录反推当前状态。

### 6.3 项目管理

- 创建、查看、编辑和归档项目；
- 项目至少记录名称、地区、行业和项目类型；
- 项目列表展示当前分析状态、风险数量和更新时间；
- 新增有效文件后现有报告自动标记 OUTDATED；
- 项目归档可恢复，归档项目默认不出现在活动列表；
- 删除操作使项目进入 30 天待删除期；
- 待删除期内允许恢复；
- 到期清除数据库业务数据和对象存储对象，同时保留不含原文的最小审计凭证。

项目名称不是唯一键。业务主键使用系统生成的 UUID。

### 6.4 文件上传与版本管理

- 一个项目支持上传多个 PDF、扫描 PDF 和 DOCX；
- 校验扩展名、MIME、文件魔数、空文件、损坏、大小、页数和压缩异常；
- 原始文件计算 SHA-256；
- 同一项目内完全重复的文件不得重复创建有效版本或重复分析；
- 跨项目保存独立业务引用，重复命中不得导致项目间共享原文或模型结果；
- 操作人员必须确认文件类型：招标文件、澄清、补遗、延期通知、附件或其他；
- 操作人员设置发布日期、生效顺序和替代关系；
- 原始文件和版本只追加，不覆盖；
- 展示文件页数、已处理页数、失败页数和处理状态；
- 不支持的格式给出明确转换建议；
- 上传采用 S3 预签名或受控分片方式；
- 对象存储区分 original、canonical、pages、artifacts、reports 和 quarantine。

### 6.5 文档解析和 OCR

- DOCX 使用固定 LibreOffice 与字体环境转换为标准化 PDF；
- 所有证据页码基于标准化 PDF；
- 每页优先提取原生文本；
- 文本质量不足时自动执行 PaddleOCR；
- 保存段落、标题、表格区域、页面坐标和阅读顺序；
- 识别目录和章节层级；
- 标记空白页、低质量页、旋转页和失败页；
- 每页保存解析器、OCR、坐标转换版本和质量分数；
- 坐标统一为左上角原点、0～1 范围的归一化矩形；
- 任一失败页面必须进入完整性计算；
- 结果可从证据定位到标准化 PDF 页面和高亮区域。

首版不要求把所有复杂表格可靠还原成完整行列数据，但必须保存表格区域和可供证据引用的文本。

### 6.6 项目权威事实

至少抽取：

- 项目名称、招标项目编号；
- 标段名称和编号；
- 招标人和招标代理机构；
- 建设地点、项目规模、招标范围、资金来源；
- 计划工期和质量标准；
- 最高投标限价；
- 评标方法和资格审查方式；
- 是否接受联合体；
- 投标截止时间；
- 开标时间和地点；
- 投标有效期；
- 投标保证金金额和形式。

每个事实必须保存候选值、来源原文、文件版本、章节、页码和坐标。多个候选不得静默覆盖。系统根据明确版本关系推荐当前值，无法确定时标记人工确认。人工修正与原始候选并存。

### 6.7 资格要求

系统对企业、资质、安全许可、企业业绩、项目经理、技术负责人、项目班子、社保、财务、信用、联合体、账户及四川平台登记等要求生成独立结构化记录。

每条要求至少包含：

- 要求对象和分类；
- 条件、操作符、阈值、单位；
- 时间范围；
- 强制资格或加分属性；
- 建议证明材料；
- 证据；
- 不确定原因和复核状态。

未接入企业资料时只允许显示“需要企业确认”或“无法判断”，不得显示“已满足”。

### 6.8 否决投标风险

- 识别明确否决和疑似高风险条款；
- 覆盖资格、形式、响应、报价、技术和程序触发条件；
- 每条风险关联触发条件、可能后果、建议动作和证据；
- 同一要求的多个证据聚合到一条风险；
- 相互冲突的条件生成冲突风险；
- CRITICAL 和 HIGH 风险执行独立验证；
- 无有效证据不得形成正式风险；
- 操作人员可确认、修订、驳回、升级或降级；
- 人工修改必须保留系统原等级和原结论。

### 6.9 评分项

- 将评标办法拆为独立评分项；
- 提取分值、满分条件、分档规则和证明材料；
- 生成评分准备清单；
- 使用 Decimal 计算已识别总分；
- 与招标文件声明总分比较；
- 无法闭合时报告标记 INCOMPLETE；
- 评分项的每个分档和证明材料均可定位证据。

客观与主观评分的自动分类不作为首版验收要求。

### 6.10 关键时间

- 提取文件获取、提疑、踏勘、保证金、投标、开标和解密时间；
- 保存 Asia/Shanghai 时区、适用标段、来源原文和解析状态；
- 识别不同章节和不同版本中的时间冲突；
- 澄清或延期生效后旧时间保留但标记失效；
- 按时间顺序展示当前有效日程；
- 模糊或不完整时间不得自动补全。

### 6.11 材料准备清单

- 根据资格、评分、承诺和格式要求生成材料清单；
- 每项材料关联用途、要求和证据；
- 标记原件、扫描件、截图、签字和盖章要求；
- 相同材料可以在展示层聚合多个用途；
- 不提供负责人、完成状态和备注协作功能。

### 6.12 技术标提示

- 提取施工组织设计章节和评分模块；
- 提取暗标格式、页数、字体、颜色和识别性信息限制；
- 输出带来源的技术标章节建议；
- 不生成完整施工组织设计；
- 工程重点难点、质量、安全、环保和季节性施工的深度语义提示不作为首版硬验收项。

### 6.13 报价提示

- 提取最高投标限价及来源；
- 提取报价唯一性、完整性和异常报价要求；
- 识别不同章节或版本的限价冲突；
- 对工程量清单文件只保存文件元数据和引用，不进行完整计价；
- 暂列金额、暂估价和不可竞争费用的完整结构化抽取不作为首版硬验收项。

### 6.14 合同与履约风险

- 合同风险必须引用具体条款；
- 输出仅使用“提示”“建议复核”等辅助措辞；
- 不输出确定性法律结论；
- 报告中展示合同风险，不提供独立合同风险页面；
- 风险分类增强和企业内部接受意见不在首版范围。

### 6.15 文件一致性与冲突

至少检查：

- 项目名称、编号和标段；
- 工期、质量标准和最高限价；
- 资格要求；
- 公告、正文、附件和版本间时间；
- 联合体要求；
- 招标文件提及但未上传的附件。

每个正式冲突至少包含两条相互矛盾的证据。能够根据显式版本关系确定时推荐当前值；不能确定时保持 UNKNOWN 并要求人工确认。

### 6.16 报告、复核和反馈

- 报告总览展示完整性、文件和页数、风险数量、资格数量、评分闭合、最近截止时间和待复核高风险数量；
- 支持按类别、风险等级、复核状态和关键词筛选；
- 点击事实、要求或风险后，在抽屉中展示标准化 PDF、页码、高亮和原文；
- 支持上一条、下一条证据导航；
- 支持单条确认、修订和驳回；
- 支持多选后的批量确认和批量驳回；
- 批量驳回必须填写统一原因，仍为每条风险创建独立 ReviewAction；
- 报告展示解析覆盖率、失败页和无法判断项；
- 发布生成不可变 ReportSnapshot；
- 发布后异步生成 PDF 并保存哈希和渲染器版本；
- 新文件进入后旧报告显示“已过期”；
- 记录确认、修订、驳回、修订原因和正确内容；
- 未经授权和脱敏的数据不得进入公共评测集。

### 6.17 发布条件

同时满足以下条件才能发布：

- AnalysisRun 状态为 READY；
- 完整性为 COMPLETE；
- 无失败或运行中的分析任务；
- 所有 CRITICAL 和 HIGH 风险已完成人工复核；
- 当前文件版本集合与分析输入一致；
- 报告快照 Schema 校验通过。

不满足条件时允许查看当前结果，但发布接口必须返回稳定错误码和未满足项列表。

## 7. 核心数据结构

### 7.1 聚合和职责

| 聚合或实体 | 职责 |
|---|---|
| Project | 一次招标分析的生命周期 |
| Document | 逻辑文件及业务类型 |
| DocumentVersion | 不可变文件版本、哈希和生效元数据 |
| DocumentRelation | 替代、补充和引用关系 |
| Page / PageBlock | 页面文本、坐标、质量和解析状态 |
| Section / Clause | 章节结构和可引用条款 |
| ProjectFactCandidate / ProjectFact | 候选事实和当前权威事实 |
| RequirementCandidate / Requirement | 候选要求和正式要求 |
| RiskCandidate / RiskFinding | 候选风险和正式风险 |
| SourceCitation | 正式数据到文件、页面和原文的证据关系 |
| AnalysisRun | 一次完整分析的编排和输入版本集合 |
| AnalysisTask / TaskAttempt | 原子任务和每次执行尝试 |
| ReviewAction | 不可变人工复核记录 |
| ReportSnapshot | 不可变发布快照 |
| OutboxEvent | 事务内任务和事件投递 |
| ModelInvocation | DeepSeek 调用版本、耗时、Token、费用和结果状态 |
| OperationLog | 不可修改的关键操作记录，不包含用户身份 |

### 7.2 通用数据约束

- 业务 ID 使用 UUID；
- 金额和分值使用 PostgreSQL NUMERIC 与 Python Decimal；
- 时间使用带时区类型，业务默认时区为 Asia/Shanghai；
- 正式实体使用关系字段表达类型、状态、金额、时间、版本和约束；
- JSONB 仅保存供应商特定元数据、模型原始结构化响应和快照展示配置；
- 不得把正式领域模型整体塞入单个 JSONB；
- 所有幂等要求同时由应用逻辑和数据库唯一约束保证；
- 删除使用显式生命周期，不依赖 ORM 隐式级联清除对象存储。

### 7.3 证据结构

SourceCitation 至少包含：

- citation_id；
- project_id；
- document_id、document_version_id；
- page_id、page_number；
- section_id、clause_id；
- 原文摘录和原文哈希；
- 归一化坐标 x、y、w、h；
- 坐标来源和转换版本；
- OCR 质量和证据验证状态。

模型不得生成 document_id、page_number 或坐标。这些字段只能通过模型返回的 clause_id 回查。

### 7.4 风险结构

RiskFinding 至少包含：

- 稳定风险编号；
- 标题、类别、等级；
- 明确风险或疑似风险属性；
- 系统原始原因、后果和建议；
- 当前人工展示结论；
- 置信度和不确定原因；
- review_status；
- 是否需要人工复核；
- 分析运行、模型、提示词、Schema 和规则版本；
- 创建和更新时间。

## 8. API 契约

### 8.1 通用规则

- 前缀为 /api/v1；
- OpenAPI 是前后端契约来源；
- 前端类型和客户端由 OpenAPI 生成；
- 列表接口统一分页、筛选和最大页大小；
- 错误响应采用统一 Problem Details 结构并包含稳定 error_code；
- 关键写接口支持 Idempotency-Key；
- API 不暴露数据库表、Celery 任务结构、模型密钥或对象存储凭证；
- 首版 API 不接受也不推导用户、角色或租户上下文；
- 项目归属校验、状态机和数据完整性校验仍必须由服务端执行。

### 8.2 资源组

| 资源组 | 主要用例 |
|---|---|
| projects | 创建、编辑、归档、恢复、删除和列表 |
| documents | 获取上传凭证、完成上传、确认类型和版本关系 |
| analysis-runs | 发起、取消、查询进度和任务详情 |
| facts | 查询候选、确认和修正权威事实 |
| requirements | 分类查询资格、评分、时间、材料和其他要求 |
| risks | 查询、筛选、单条复核和批量复核 |
| pages | 页面元数据、授权 PDF 地址和证据定位 |
| reports | 发布、查看版本和获取 PDF |
| operation-logs | 查询不包含用户身份的关键操作记录 |

### 8.3 上传接口

推荐流程：

1. 前端提交文件名、大小、声明 MIME；
2. 后端校验部署限额，创建上传会话；
3. 后端返回短期预签名上传信息；
4. 前端直传 S3 兼容存储；
5. 前端通知上传完成；
6. 后端校验对象大小、文件魔数和哈希后正式创建 DocumentVersion。

上传完成前的对象不属于正式业务文件。

### 8.4 进度接口

- SSE 提供分析状态和任务摘要变化；
- 同时提供可轮询的权威快照接口；
- SSE 只传递 ID、状态和版本号，不传整页文本；
- 前端收到事件后刷新 TanStack Query；
- 断线重连不影响业务状态。

### 8.5 批量复核接口

请求必须包含：

- 明确 risk_ids；
- 目标动作；
- 原因或备注；
- 客户端幂等键；
- 每条风险的期望版本号。

响应必须逐条返回处理结果。发生并发修改时只拒绝冲突项，不覆盖较新的人工结论。

## 9. AI、规则和检索

### 9.1 DeepSeek 接入

- 通过 ModelGateway 端口接入 DeepSeek；
- DeepSeek 适配器使用其 OpenAI-compatible Chat Completions 协议；
- Base URL、API Key、经济模型、强模型、超时、并发和预算均通过环境变量配置；
- 具体模型名称不得写入领域代码；
- 使用 JSON 输出，并继续执行 JSON 解析、Pydantic Schema、条款 ID、证据原文和领域规则校验；
- JSON 空响应、截断、Schema 错误和限流必须显式分类；
- 原始请求响应进入受限存储，普通日志只记录哈希和元数据；
- 单次请求只发送原子任务必需的条款；
- 不允许把整本招标文件一次性发送给模型。

参考：

- DeepSeek API First Call：https://api-docs.deepseek.com/guides/function_calling/
- DeepSeek JSON Output：https://api-docs.deepseek.com/guides/json_mode/

### 9.2 模型任务分级

- 经济模型：章节分类、明确字段、材料类型和显式要求抽取；
- 强模型：隐式否决、复杂评分、合同风险、跨条款冲突候选和独立验证；
- 模型路由由配置和任务类型确定，不允许业务模块直接选择供应商 SDK。

### 9.3 确定性规则

以下任务不得主要依赖模型：

- SHA-256 和重复文件；
- 页数和页面完整性；
- 金额、日期和分值；
- 评分总分闭合；
- 项目编号和结构化字段精确比较；
- 文件版本关系；
- 状态转换、项目归属和版本关系。

四川规则放在独立 sichuan_construction 规则包，具有 Manifest、适用范围、版本、依据来源和兼容 Schema 版本。

### 9.4 全量分析

“找出全部资格、否决和时间要求”必须全量扫描条款，不允许用 Top-K 检索代替。首版冲突候选以结构化字段、全文检索和确定性规则为主；核心 P0 不依赖外部 Embedding 服务。

## 10. UI/UX 规格

### 10.1 总体要求

- 仅支持桌面端；
- 以 1440px 宽度作为主要设计基准；
- 低于最小支持宽度时提示使用桌面端，不制作移动端布局；
- 风格简约、克制、信息密度适中；
- 高风险不得只用颜色表达，必须同时使用文字、图标和等级；
- 系统结论与人工结论视觉区分；
- 不展示虚假进度；
- 失败必须可见、可定位和可操作。

### 10.2 页面

首版页面：

- 项目列表；
- 创建和编辑项目；
- 文件管理；
- 分析进度；
- 报告总览；
- 资格要求；
- 否决风险；
- 评分矩阵；
- 时间线；
- 材料、技术、报价和合同报告分区；
- 文档阅读器；
- 报告发布与 PDF 下载；
- 关键操作记录。

合同风险在报告分区展示，不建设独立路由页面。

### 10.3 证据抽屉

- 点击事实、要求、风险或评分项后从右侧打开证据抽屉；
- 抽屉内展示文件名、版本、章节、页码、原文和 PDF 页面；
- PDF 高亮使用归一化坐标；
- 支持缩放、旋转、上一条和下一条证据；
- 展示 OCR 质量和解析状态；
- 抽屉关闭后保留列表筛选和滚动位置；
- 可通过独立文档阅读器继续查看完整文件。

### 10.4 批量复核

- 列表行支持多选；
- 只能选择当前状态允许复核的风险；
- 支持批量确认和批量驳回；
- 批量驳回要求填写原因；
- 逐条修订字段不支持批量操作；
- 操作前展示数量、目标状态和不可处理项；
- 操作后显示逐条结果和失败原因。

## 11. 异常和边界条件

### 11.1 文件异常

- 扩展名、MIME 和文件魔数不一致：拒绝并审计；
- 空文件、损坏文件、加密且无法读取的文件：拒绝；
- 超过部署字节限制或项目总页数超过 500：拒绝继续纳入分析；
- 恶意文件检查未完成：文件停留在 quarantine，不进入解析；
- 同项目相同哈希：提示重复，不重复分析；
- DOCX 转换失败：保留原件，标记失败并给出人工转换建议；
- 单页解析失败：保留其他页面结果，完整性为 INCOMPLETE。

### 11.2 AI 异常

- 超时、网络错误和限流：指数退避重试；
- JSON 空响应、截断或 Schema 不合法：有限重试；
- 条款 ID 或证据不合法：不得写入正式结果；
- 连续失败：任务 FAILED，保留已完成结果，分析不进入 READY；
- 不允许静默切换到其他供应商或模型；
- 重新执行必须生成新的 TaskAttempt，不覆盖失败记录。

### 11.3 并发和幂等

- 同一项目同一输入版本集合只能存在一个活动分析运行；
- 相同任务幂等键不得产生重复正式结果；
- Outbox 允许重复投递，Worker 必须幂等消费；
- 人工复核使用乐观版本检查；
- 报告发布使用幂等键，相同快照重复请求返回同一发布结果。

### 11.4 取消

- 用户取消后 AnalysisRun 先进入 CANCEL_REQUESTED；
- 未领取任务直接取消；
- 正在运行的任务在安全检查点协作停止；
- 已完成的解析和分析产物保留，可供审计和后续新运行复用；
- 取消中的运行不得发布；
- 取消不删除原始文件。

### 11.5 过期与重分析

- 新有效文件或替代关系变更后，当前报告立即 OUTDATED；
- 历史报告内容和 PDF 不修改；
- 新分析运行引用新的明确文件版本集合；
- 模型、提示词、Schema、解析器或规则升级不会后台静默改写历史结果；
- 用户显式发起新分析后才生成新业务结果。

## 12. 非功能要求

### 12.1 性能

- 单项目支持最多 500 页；
- 普通查询 P95 目标不超过 2 秒；
- 进度状态端到端更新目标不超过 10 秒；
- 上传与分析完全分离；
- OCR、普通解析和 API 使用独立进程或容器；
- 任务并发度、超时和限流可配置；
- 不承诺固定分析完成时长，界面只展示基于真实任务的进度。

### 12.2 可用性和恢复

- 原始文件成功接入后不得因分析失败丢失；
- 服务重启后从 PostgreSQL 恢复业务状态；
- Outbox 补偿未投递任务；
- 心跳扫描恢复卡死任务；
- Redis 不作为唯一业务状态来源；
- 报告快照、原始文件和标准化 PDF纳入备份；
- 首版目标月度可用性 99.5%。

### 12.3 安全和隐私

- 全链路 TLS；
- 对象默认私有，通过短期授权地址访问；
- 密钥不进入前端、代码、镜像和日志；
- 对象存储启用服务端加密；
- 模型请求只发送任务所需条款；
- 普通日志不记录完整原文；
- 上传、查看、下载、删除、分析、取消、重试、复核和发布均记录关键操作；
- 客户文档和复核数据不得用于模型训练，原文或模型结果不得跨项目缓存复用；
- 数据删除可审计。

### 12.4 可维护性

- 模块化单体，不拆微服务；
- 领域层不依赖 FastAPI、Celery、SQLAlchemy 或模型 SDK；
- 路由和 Celery Task 不承载业务逻辑；
- 文件解析、AI、对象存储和规则通过端口与适配器替换；
- 状态转换集中定义；
- 所有模块提供职责、入口、状态和依赖说明；
- 不创建全局巨型 services、models 或 utils；
- 新增和修改代码使用必要的多行简体中文注释解释业务边界和非显然设计原因。

### 12.5 可观测性

日志、指标和 Trace 至少关联：

- trace_id；
- project_id；
- document_version_id；
- analysis_run_id；
- analysis_task_id；
- model_invocation_id。

记录 API 延迟、文件处理率、OCR 比例、任务重试、DeepSeek Token 和费用、Schema 失败、风险复核、报告生成及基础设施状态。

## 13. 部署、迁移和回滚

### 13.1 Linux 部署基线

单台 Linux 服务器使用 Docker Compose 运行：

- frontend；
- nginx；
- api；
- worker-default；
- worker-ocr；
- scheduler；
- postgres；
- redis；
- minio；
- otel-collector。

DeepSeek 是外部服务。API、Worker 和 OCR Worker 使用不同镜像或依赖层，API 镜像不得包含不必要的 OCR/GPU 运行时。

### 13.2 配置

环境变量至少覆盖：

- 数据库、Redis、S3/MinIO 连接；
- DeepSeek Base URL、API Key、模型名称、超时和并发；
- 文件字节数和数量限制；
- 对象授权 URL 有效期；
- 日志、指标和备份位置。

启动时执行配置 Schema 校验，缺少关键配置时必须失败退出。

### 13.3 数据迁移

- 本系统从空库首次部署，不迁移旧系统数据；
- Schema 使用 Alembic 管理；
- 必须支持从空库创建当前版本；
- 每次发布先备份 PostgreSQL 和对象存储关键数据；
- 迁移采用显式、可审计步骤；
- 不在应用启动请求路径中隐式执行迁移。

### 13.4 发布与回滚

- 应用镜像不可变并带版本号；
- 发布前停止接收新的分析运行，允许短任务结束或安全取消；
- 数据库迁移默认只前进；
- 无 Schema 冲突时，应用通过切换到上一镜像回滚；
- 涉及破坏性 Schema 变更时，发布前必须生成恢复点；
- 迁移失败时停止新版本，不继续启动 API 和 Worker；
- 必要时恢复数据库和对象存储到同一恢复点；
- 恢复后 Scheduler 根据数据库状态补偿未完成任务；
- Redis 队列不得作为恢复依据。

首版不设自动发布门禁，发布动作由运维人员人工执行并核对检查清单。

## 14. 测试策略

### 14.1 原则

- 最终验收由用户人工完成；
- 系统提供清晰、逐项可验证的功能清单；
- AI 召回率、证据正确率等指标可以记录，但暂不阻断发布；
- 实施过程中仍编写必要的单元、集成、契约、解析回归和任务恢复测试；
- 不自动启动浏览器测试，UI 由用户手工测试。

### 14.2 实施期测试

- 领域单元测试：状态机、金额、分值、日期、版本、项目归属和发布条件；
- 应用测试：事务、幂等、失效传播和跨模块用例；
- 集成测试：真实 PostgreSQL、Redis 和 S3 兼容存储；
- Worker 测试：Outbox、重试、心跳、取消和恢复；
- 契约测试：OpenAPI、Celery Payload 和模型 Schema；
- 文档回归：固定 PDF、扫描 PDF、DOCX、失败页和坐标；
- 前端组件测试：筛选、状态、批量复核和证据抽屉；
- UI 端到端验收：由用户手工执行。

## 15. 人工功能验收清单

以下清单是首版最终验收依据。

### 15.1 运行边界和应用入口

- [ ] 系统从空库启动时不创建 Organization、User、Membership、Role、Session 或 Token 数据；
- [ ] 进入应用后直接展示项目列表，不出现登录、注册、成员或角色页面；
- [ ] 业务 API 不要求 Bearer Token、身份 Cookie 或用户/组织请求头；
- [ ] 数据库业务表不包含 organization_id、user_id、created_by 或 reviewed_by 等身份字段；
- [ ] 所有可访问应用入口的操作人员能够使用同一套业务功能；
- [ ] 生产部署文档明确要求受控内网、VPN 或等价可信网络入口，且内部基础设施端口不公开；
- [ ] 关键操作记录包含请求、资源、时间和结果，但不伪造操作者身份。

### 15.2 项目和文件

- [ ] 可创建、编辑、归档和恢复项目；
- [ ] 项目列表展示状态、风险数量和更新时间；
- [ ] 一个项目可上传多个 PDF、扫描 PDF 和 DOCX；
- [ ] 空文件、损坏文件、伪造类型和超限文件被明确拒绝；
- [ ] 同项目重复文件不重复分析；
- [ ] 原文件和历史版本不可覆盖；
- [ ] 用户可确认文件类型、生效顺序和替代关系；
- [ ] 项目超过 500 页时系统阻止纳入分析；
- [ ] 文件页数、成功页数、失败页数和状态可查看。

### 15.3 解析和证据

- [ ] 原生 PDF 能提取逐页文本；
- [ ] 扫描 PDF 自动进入 OCR；
- [ ] DOCX 能生成可复现的标准化 PDF；
- [ ] 旋转、空白、低质量和失败页面有明确状态；
- [ ] 失败页面使报告显示不完整；
- [ ] 结果可定位到正确文件、版本、页码和高亮区域；
- [ ] 缩放和旋转后高亮仍与原文对应；
- [ ] 模型不能伪造页码、文件或坐标。

### 15.4 分析结果

- [ ] 系统生成项目权威事实并展示候选和证据；
- [ ] 冲突事实不会被静默覆盖；
- [ ] 系统生成资格要求和建议材料；
- [ ] 无企业资料时不显示“已满足”；
- [ ] 系统区分明确否决与疑似风险；
- [ ] CRITICAL/HIGH 风险执行独立验证；
- [ ] 系统生成评分矩阵并核对总分；
- [ ] 分值不闭合时报告不完整；
- [ ] 系统生成关键时间线并处理延期失效；
- [ ] 系统生成材料、技术、报价和合同提示；
- [ ] 系统识别项目名称、编号、工期、限价、资格和时间冲突；
- [ ] 冲突风险至少展示两条证据；
- [ ] 所有正式风险都有有效证据；
- [ ] DeepSeek 最终失败时不会产生无依据正式结论。

### 15.5 复核和报告

- [ ] 可确认、修订和驳回单条风险；
- [ ] 系统原结果和人工结果同时保留；
- [ ] 可多选风险并批量确认；
- [ ] 可多选风险并填写原因后批量驳回；
- [ ] 批量操作展示逐条结果和冲突项；
- [ ] 点击结果可在抽屉中查看 PDF 证据；
- [ ] 未解析页面或失败任务存在时不能发布；
- [ ] 未复核的 CRITICAL/HIGH 风险存在时不能发布；
- [ ] 满足条件后可发布不可变报告；
- [ ] 可生成并下载 PDF 报告；
- [ ] PDF 包含完整性、失败页、待确认项和免责声明；
- [ ] 新有效文件进入后旧报告立即显示过期；
- [ ] 历史报告和 PDF 内容不被重写。

### 15.6 异常、恢复和安全

- [ ] 外部 API 临时失败会重试并保留尝试记录；
- [ ] Worker 重启后任务可从数据库状态恢复；
- [ ] 重复任务消息不会生成重复业务结果；
- [ ] 用户取消后未开始任务停止，已完成产物保留；
- [ ] 对象存储文件默认不可公开访问；
- [ ] API、Worker、数据库、Redis 和对象存储内部端口不直接暴露到公共网络；
- [ ] DeepSeek、数据库和存储密钥不暴露给前端；
- [ ] 普通日志不含密码、Token、签名 URL 和完整敏感原文；
- [ ] 项目删除进入 30 天待删除期并可恢复；
- [ ] PostgreSQL、对象存储和报告可以按备份流程恢复。

## 16. 需求完成定义

首版 MVP 只有在以下条件同时满足时才视为完成：

- 本文范围内功能已经实现；
- 三类输入文件能在不超过 500 页的项目中完成主流程；
- 所有正式结果符合证据规则；
- 项目归属、状态转换、证据规则和发布条件由后端强制执行；
- Linux 环境部署成功；
- 用户完成第 15 章人工验收并确认结果；
- 未完成项被明确记录，不以隐藏错误或假进度代替。

## 17. 非阻塞部署参数

以下参数不阻塞开发，由部署时配置：

- Linux 发行版和服务器规格；
- 域名和 TLS 证书；
- 单文件字节数、项目文件数和项目总字节数；
- DeepSeek 当前有效模型名称；
- DeepSeek 并发、超时和单项目预算；
- MinIO 本地磁盘路径和备份目标；
- 30 天删除任务的具体执行时间。

这些参数必须通过强类型配置进入系统，不得散落为魔法常量。
